# 文件防篡改校验

桌面客户端如何对随包分发的文件做防篡改校验。下文将被保护的产物分为两类：**可执行文件（Windows 的 PE exe/dll、macOS 的 Mach-O）** 与 **普通资源（配置 / 静态资源 / 脚本）**，并给出 Windows 与 macOS 的对应实现。

---

## 核心原则：signature-first，而非 hash-first

> **以「数字签名验证」为第一判定，而不是「整文件哈希为主、签名做兜底」。**

整文件哈希与已签名的可执行文件天生不合，尤其在 Windows 上：**Authenticode 签名并不覆盖整个文件**——PE 头的 `CheckSum`、数据目录里的证书表项、文件末尾的签名块都不参与签名哈希。杀毒软件 / EDR / 系统装载器会改写这些「非代码区」（重算校验和、打标记、留痕），导致：

- **整文件 SHA-256 变了**（这些字节确实变了）；
- 但**被执行的代码没变**，数字签名**依旧有效**。

因此「hash 为主 + 签名兜底」从一开始就在制造误报，然后再打补丁。反过来以签名为主，就没有这个空档。

> macOS 的机制不同（见下文）：在启用 hardened runtime / library validation，或经 Gatekeeper 校验的场景下，改动已签名 Mach-O 的代码页会破坏签名并被系统拒绝加载。加之 macOS 上少有杀软去改写已签名二进制，因此「非代码区被改」的整文件哈希误报在 Mac 下基本不出现。但 signature-first 依旧是更强、更省心的判定方式。

---

## 方案设计（平台无关的骨架）

验签层抽象成一个统一接口，返回三档结果：**trusted（放行）/ untrusted（篡改）/ unavailable（无法判定）**。上层的判定分级与降噪逻辑跨平台保持一致，只有「验签实现」和「Pinning 方式」按平台落地。

签名验证虽是最强判定，但在部分机器上会 **unavailable**（系统验签组件被 EDR 限制、超时、证书链在该机无法验证、离线吊销检查失败等）。因此不靠单一手段，而是**分层多信号**——任一层给出明确结论即采纳，逐层填补上一层的空档：

```
1) 验签（signature-first）
     能判定 → trusted / untrusted 直接采纳（最强，含签名者身份）
     unavailable ↓
2) 离线 Authenticode 规范化哈希 vs 清单（不依赖验签环境，几乎总能执行）
     匹配   → 受签名保护的内容未变，放行
     不匹配 → 可疑 ↓
3) 仍无明确结论
     → 重试 + 多次采样降噪，连续 N 次才告警
```

**最重要的一条铁律：`unavailable`（没能完成判定）绝不等于篡改。** 只有 untrusted（判定为无效 / 签名者不符）或哈希明确失配才是真信号；环境原因导致的无法判定必须走降噪，而非直接告警。

### 1. 可执行文件：只验签名，不比整文件哈希

- 验证签名结构完整、未被破坏；
- 验证证书链可回溯到受信任根。

签名有效 → 放行；签名无效 / 被去签名 / 结构破坏 → 判篡改。改动「非代码区」不影响签名有效性，**误报从架构上消失**。

| | Windows | macOS |
| --- | --- | --- |
| 目标格式 | PE（exe/dll） | Mach-O |
| 原生 API | `WinVerifyTrust` | Security.framework：磁盘文件用 `SecStaticCodeCheckValidity`，运行中代码用 `SecCodeCheckValidity` |
| 命令行 | `signtool verify /pa` | `codesign --verify --deep --strict`、`spctl -a` |

### 2. 叠加 Pinning，保证「确实是我签的」

只验「链到受信根」不够——攻击者用任意一张可信 CA 签发的证书重签就能通过。必须再约束签名者身份：

- **Windows**：比对签名者证书的**公钥指纹**是否等于内置的期望值。Pin 公钥而非证书序列号 / 指纹的好处是不依赖有效期等易变字段；但要注意代码签名证书**续期 / 换 CA 时通常会生成新密钥对**，公钥随之变化，届时需同步更新内置期望值（除非刻意复用同一密钥对）。若想降低更新频率，可改 pin 中间 / 根证书 + subject 名。
- **macOS**：用系统的 **Designated Requirement（代码签名要求语言）** 约束签名者身份，例如固定 **Team ID**：`anchor apple generic and certificate leaf[subject.OU] = "TEAMID"`；更严谨可再加 Developer ID 中间证书 OID：`and certificate 1[field.1.2.840.113635.100.6.2.6] exists`。这是苹果官方推荐方式，比手动比对公钥更地道。（`cdhash` 钉死的是某个具体构建、每次发版都变，属于「锁定特定二进制」而非「签名者身份」，用途不同。）

签名有效 **且** 身份约束命中 → 放行；签名者不符 → 判篡改。

### 3. 验签不可用时的兜底：离线 Authenticode 规范化哈希

验签 `unavailable` 时，需要一个**不依赖验签环境**的判断来回答“被执行的内容到底变没变”。这里**不要用整文件哈希**（会被杀软改的非代码区带偏，正是最初的误报源），也**不要只算 `.text`**（会漏掉导入表 / 入口点 / TLS 回调 / `.rdata` 等，导致漏报）。

正确做法是复算 **Authenticode 规范化哈希（PE image hash）**——即“**排除 `CheckSum` / 证书表项 / 尾部签名块这三处后，其余全部内容的哈希**”，覆盖面正好等于签名保护的范围：

- 构建期离线算出该规范化哈希，写进清单（清单本身用离线私钥签名，见下节）；
- 运行期验签 `unavailable` 时，客户端**自行复算**并与清单比对：匹配 → 内容未变，放行；不匹配 → 进入降噪判定。

它只依赖“读文件 + 按 PE 规则算哈希”，不依赖 PowerShell / 系统验签服务 / 证书链 / 网络吊销检查，几乎在任何机器上都能跑出结论。代价是需要按 Authenticode 规则解析 PE（跳过三处、处理对齐），属一次性工程成本。

> 若不愿承担该复杂度，可省略本层，让 `unavailable` 直接进入下面的降噪判定——实现最简、零误报风险，代价是对“真篡改且恰好验签也不可用”的机器判断力较弱。
>
> macOS 一般无需本层：Code Directory 本就是按页哈希代码（cdhash），系统验签已内建等价能力。

### 4. 普通资源：离线签名的哈希清单（跨平台通用）

配置、静态资源、脚本等无法做代码签名，改用哈希清单：

- 构建期生成 `SHA-256` 清单；
- **清单整体用离线私钥（如 Ed25519）签名**，客户端内置公钥先验清单签名，再逐文件比对哈希。

这类文件不会被杀软改「非代码区」，整文件哈希对它们是可靠的。在 macOS 上，若资源已打进 `.app` bundle 并参与整体签名，其完整性已由系统签名（`_CodeSignature/CodeResources`）覆盖，可省去自建清单。

### 5. 「无法判定」的降噪（跨平台通用）

走完前面几层仍无明确结论（`unavailable` 且规范化哈希也未能放行）时，**仍不等同于篡改**：

- 先做带退避的重试（2~3 次）；
- 仍失败则**暂存 + 多次采样**：连续 N 次仍无法判定才告警，过滤偶发环境噪声。

## 平台对照

| 方案要素 | Windows | macOS |
| --- | --- | --- |
| 核心思路（signature-first + 身份约束） | ✅ | ✅ |
| 「整文件 hash 误报」前提 | 严重（杀软改非代码区） | 基本不存在 |
| 验签实现 | `WinVerifyTrust` / `signtool` | `SecStaticCodeCheckValidity` / `codesign` / `spctl` |
| 身份约束方式 | 公钥指纹 Pinning | Designated Requirement（Team ID / Developer ID） |
| 验签不可用时的兜底 | 离线 Authenticode 规范化哈希 | 一般无需（cdhash 已内建） |
| 普通资源 | Ed25519 签名的哈希清单 | 同左；bundle 内资源可复用系统签名 |
| 无法判定降噪 | 重试 + 多次采样 | 同左 |

---

## 一句话总结

**分层多信号，且 `unavailable` 绝不等于篡改**：验签（signature-first）+ 签名者身份约束为主 → 验签不可用时用离线 Authenticode 规范化哈希兜底 → 仍无解则重试 + 多次采样降噪；普通资源用离线签名的哈希清单。骨架跨平台通用，Windows 用 `WinVerifyTrust` + 公钥 Pinning，macOS 用 Security.framework + Designated Requirement。从根上避免整文件哈希的误报，任一单点失效都有下一层接住。
