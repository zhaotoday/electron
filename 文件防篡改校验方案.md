# 文件防篡改校验

桌面客户端如何对随包分发的文件做防篡改校验。Windows 与 macOS 的代码签名机制差异很大，**两端分开独立设计，不做统一抽象 / 兼容层**。被保护的产物分两类：**可执行文件**（Windows 的 PE exe/dll、macOS 的 Mach-O）与 **普通资源**（配置 / 静态资源 / 脚本）。

两端共同遵循两条原则：

- **signature-first**：以数字签名验证为第一判定，而不是「整文件哈希为主、签名做兜底」。
- **`unavailable` ≠ 篡改**：只有「判定为签名无效 / 签名者不符」或「哈希明确失配」才是真信号；因环境原因**没能完成判定**时必须走降噪，绝不直接告警。

---

## 一、Windows 方案

### 为什么不用整文件哈希

**Authenticode 签名并不覆盖整个文件**——PE 头的 `CheckSum`、数据目录里的证书表项、文件末尾的签名块都不参与签名哈希。杀毒软件 / EDR / 系统装载器会改写这些「非代码区」（重算校验和、打标记、留痕），导致：

- **整文件 SHA-256 变了**（这些字节确实变了）；
- 但**被执行的代码没变**，数字签名**依旧有效**。

所以「hash 为主 + 签名兜底」从一开始就在制造误报。反过来以签名为主，就没有这个空档。

### 分层判定链

签名验证是最强判定，但在部分机器上会 `unavailable`（系统验签组件被 EDR 限制、超时、证书链在该机无法验证、离线吊销检查失败等）。因此采用**分层多信号**，任一层给出明确结论即采纳：

```
1) 验签（WinVerifyTrust + 公钥 Pinning）
     能判定 → trusted / untrusted 直接采纳（最强，含签名者身份）
     unavailable ↓
2) 离线 Authenticode 规范化哈希 vs 清单（不依赖验签环境，几乎总能执行）
     匹配   → 受签名保护的内容未变，放行
     不匹配 → 可疑 ↓
3) 仍无明确结论
     → 重试 + 多次采样降噪，连续 N 次才告警
```

### 1. 验签

- 用 `WinVerifyTrust`（原生 API）验证签名结构完整、证书链可回溯到受信任根；命令行等价手段为 `signtool verify /pa`。
- 签名有效 → 放行；签名无效 / 被去签名 / 结构破坏 → 判篡改。改动「非代码区」不影响签名有效性，**误报从架构上消失**。

### 2. 公钥 Pinning

只验「链到受信根」不够——攻击者用任意一张可信 CA 签发的证书重签就能通过。再比对签名者证书的**公钥指纹**是否等于内置期望值：

- Pin 公钥而非证书序列号 / 指纹，不依赖有效期等易变字段；
- 但代码签名证书**续期 / 换 CA 时通常会生成新密钥对**，公钥随之变化，届时需同步更新内置期望值（除非刻意复用同一密钥对）；
- 想降低更新频率，可改 pin 中间 / 根证书 + subject 名。

签名有效 **且** pin 命中 → 放行；签名者不符 → 判篡改。

### 3. 验签不可用时的兜底：离线 Authenticode 规范化哈希

验签 `unavailable` 时，需要一个**不依赖验签环境**的判断来回答“被执行的内容到底变没变”。这里**不要用整文件哈希**（会被非代码区带偏，正是最初的误报源），也**不要只算 `.text`**（会漏掉导入表 / 入口点 / TLS 回调 / `.rdata` 等，导致漏报）。

正确做法是复算 **Authenticode 规范化哈希（PE image hash）**——即“**排除 `CheckSum` / 证书表项 / 尾部签名块这三处后，其余全部内容的哈希**”，覆盖面正好等于签名保护的范围：

- 构建期离线算出该规范化哈希，写进清单（清单本身用离线私钥签名，见下节）；
- 运行期验签 `unavailable` 时，客户端**自行复算**并与清单比对：匹配 → 内容未变，放行；不匹配 → 进入降噪判定。

它只依赖“读文件 + 按 PE 规则算哈希”，不依赖 PowerShell / 系统验签服务 / 证书链 / 网络吊销检查，几乎在任何机器上都能跑出结论。代价是需要按 Authenticode 规则解析 PE（跳过三处、处理对齐），属一次性工程成本。

> 若不愿承担该复杂度，可省略本层，让 `unavailable` 直接进入降噪——实现最简、零误报风险，代价是对“真篡改且恰好验签也不可用”的机器判断力较弱。

### 4. 普通资源：离线签名的哈希清单

配置、静态资源、脚本等无法做代码签名，改用哈希清单：

- 构建期生成 `SHA-256` 清单；
- **清单整体用离线私钥（如 Ed25519）签名**，客户端内置公钥先验清单签名，再逐文件比对哈希。

这类文件不会被杀软改「非代码区」，整文件哈希对它们是可靠的。

### 5. 「无法判定」的降噪

走完前面几层仍无明确结论（`unavailable` 且规范化哈希也未能放行）时，**仍不等同于篡改**：

- 先做带退避的重试（2~3 次）；
- 仍失败则**暂存 + 多次采样**：连续 N 次仍无法判定才告警，过滤偶发环境噪声。

## 二、macOS 方案

macOS 用 **Mach-O + 代码签名**，机制与 Windows 完全不同，方案也更简单：

- 签名信息在 `LC_CODE_SIGNATURE` 指向的 `__LINKEDIT` 区域，其 **Code Directory 存放代码各页的哈希**，而对 Code Directory 结构本身求的摘要即 **cdhash**（该二进制的唯一标识）；
- 在启用 hardened runtime / library validation，或经 Gatekeeper 校验的场景下，改动已签名 Mach-O 的代码页会破坏签名并被系统拒绝加载；
- 加之 macOS 上少有杀软去改写已签名二进制，**「非代码区被改」的整文件哈希误报在 Mac 下基本不出现**。

因此 macOS **不需要 Windows 那套「规范化哈希兜底层」**——系统验签已内建等价的按页代码哈希能力。

### 1. 验签 + 身份约束

- 用 Security.framework 的 `SecStaticCodeCheckValidity`（磁盘文件；运行中代码用 `SecCodeCheckValidity`）做验证；命令行等价手段为 `codesign --verify --deep --strict`、`spctl -a`。
- 身份约束用系统的 **Designated Requirement（代码签名要求语言）**，例如固定 **Team ID**：

  ```
  anchor apple generic and certificate leaf[subject.OU] = "TEAMID"
  ```

  更严谨可再加 Developer ID 中间证书 OID：`and certificate 1[field.1.2.840.113635.100.6.2.6] exists`。这是苹果官方推荐方式，比手动比对公钥更地道。

> `cdhash` 钉死的是某个具体构建、每次发版都变，属于「锁定特定二进制」而非「签名者身份」，用途不同，不用它做身份约束。

### 2. 普通资源

若资源已打进 `.app` bundle 并参与整体签名，其完整性已由系统签名（`_CodeSignature/CodeResources`）覆盖，无需自建清单。对未纳入 bundle 签名的资源，再用与 Windows 相同的 **Ed25519 签名哈希清单**。

### 3. 「无法判定」的降噪

与 Windows 一致：验签因环境原因跑不出结论时**不等于篡改**，先重试、再暂存多次采样，连续 N 次才告警。

---

## 一句话总结

- **Windows**：`WinVerifyTrust` + 公钥 Pinning 为主 → 验签不可用时用离线 Authenticode 规范化哈希兜底 → 仍无解则重试 + 多次采样降噪；普通资源用 Ed25519 签名哈希清单。
- **macOS**：`SecStaticCodeCheckValidity` + Designated Requirement 即可，无需规范化哈希兜底；bundle 内资源复用系统签名，其余用 Ed25519 清单；同样重试 + 多次采样降噪。
- 两端共识：**signature-first**，且 **`unavailable` 绝不等于篡改**。
